# SEBI Algo Trading Compliance (April 2026)

This document maps SEBI's April 2026 algorithmic trading rules for retail clients to specific implementations in this codebase. Read alongside the SEBI circular for the authoritative text.

> **Disclaimer.** This is an engineering reference, not legal advice. SEBI rules evolve; always verify current requirements with your broker before going live.

## The five things that matter

### 1. Static IP whitelisting

**Rule:** Each algo trading API connection must originate from a single static IP, registered with the broker in advance.

**Implementation:**
- A NAT Gateway with a persistent Elastic IP (`infra/modules/network/main.tf`) is the single egress point for all outbound traffic to the broker.
- Every execution node lives in the private subnet whose route table sends `0.0.0.0/0` through this NAT GW.
- The EIP is tagged `DO NOT release` in Terraform — releasing it would invalidate the whitelist with the broker.

**Operational note:** Run `terraform output whitelist_ip` after `apply` and register that IP with your broker BEFORE activating any account. The broker's API app dashboard (Kite Connect: kite.trade/connect/console) is where you set this.

**What we did NOT do:** assign one EIP per execution node (Pattern C). That's necessary only if you're trading multiple broker accounts simultaneously and need isolated rate limits per account. For single-account personal use, a single shared NAT EIP is the simpler choice (Pattern A).

### 2. Daily 2FA / TOTP authentication

**Rule:** Algo API access requires daily fresh authentication with two factors, one of which must be TOTP.

**Implementation:**
- `services/auth_refresh/handler.py` is an AWS Lambda that performs the four-step Kite login (username/password → TOTP → request_token → access_token).
- Triggered by EventBridge cron `cron(15 3 ? * MON-FRI *)` — that's 03:15 UTC = **08:45 IST**, before market open at 09:15 IST.
- Fresh `access_token` written to Secrets Manager at `valgo/kite/access_token`.
- Ingestor and execution nodes pull the token at startup; the Lambda's SNS notification confirms success.

**Why Lambda instead of a sidecar:** the auth flow runs once a day. Cold start overhead doesn't matter. Lambda is cheaper and easier to schedule than a long-running container.

**TOTP seed handling:** stored in Secrets Manager at `valgo/kite/totp_seed`, never in code or .env. Generated by the broker — Kite gives you a base32 string when you enable TOTP.

### 3. Rate limit: 10 orders per second per API key

**Rule:** Maximum 10 orders/second per API key, unless the exchange has separately approved a higher limit.

**Implementation:** `services/execution_router/rate_limiter.py`
- Redis-backed fixed-window counter keyed on `rate:orders:{account_id}`
- 1-second TTL — when the second rolls, the key expires and the counter resets
- Atomic `INCR` + conditional `EXPIRE` (NX flag) so concurrent router pods can't race
- Returns 429 to the caller when cap is exceeded

**Why fixed-window not token-bucket:** SEBI's rule is "max 10 in any 1 second". Token bucket would allow burst > 10 in a sub-second window if tokens accumulated. Fixed-window matches the rule semantics exactly.

**Edge case:** at second-boundary, two adjacent windows can each hold 10 orders → ~20 in a wall-clock second straddling the boundary. This is how brokers enforce it server-side too, so we mirror their behavior.

### 4. MARKET → MPP conversion

**Rule:** MARKET orders for retail must be converted to Market Price Protection (MPP) — a market order with a price band to protect against extreme slippage.

**Implementation:** `services/execution_node/broker_adapter.py`
- Internal `OrderType.MARKET` is passed to Kite as `ORDER_TYPE_MARKET`.
- Kite Connect's API enforces MPP server-side post-April-2026 — we don't need to set the price band; Kite caps it automatically using the exchange-defined protection range.
- For brokers that require client-side MPP price calculation (some don't), set `price = last_traded_price * (1 + protection_pct)` for BUY and `* (1 - protection_pct)` for SELL.

### 5. Audit trail

**Rule:** Every algo-generated order must be auditable — entry signal, decision rationale, risk-check outcome, broker response.

**Implementation:**
- `audit` DynamoDB table (`infra/modules/data/main.tf`) stores every event
- `services/common/dynamodb.append_audit_event` is called from:
  - Execution router on rejected pre-trade orders (`order_rejected_pretrade`)
  - Webhook handler on every inbound signal (`webhook_signal_received`)
  - Execution node on broker submission and fills (TODO: wire this up)
- Events keyed by `event_id` (UUID) with `timestamp` for time-range queries
- PITR enabled in prod for forensic restore

**What's stored per event:**
```json
{
  "event_id": "uuid",
  "timestamp": "2026-05-01T09:23:45Z",
  "event_type": "order_rejected_pretrade",
  "actor": "strategy:s1",
  "payload": { "reason": "rate_limited", "request": {...} }
}
```

## Other things SEBI cares about (lower priority, partially implemented)

### Kill switch

Not strictly mandated, but exchanges expect operators to be able to halt all algo activity within seconds.

**Implementation:**
- Redis flag `risk:kill_switch` checked on every order in the router (`services/execution_router/risk.py`)
- Toggleable from the admin panel (Risk section) or CLI
- Test weekly: see `docs/runbook.md` → "Things to never do"

### Daily P&L cap

**Implementation:**
- `risk:daily_pnl:{account_id}` Redis key, updated as fills come in
- `RiskLimits.max_daily_loss` enforced in `risk.check()` — denies new orders when threshold breached
- TTL of 24h so it self-resets

### Position limits

**Implementation:**
- `risk:open_positions:{account_id}` Redis SET tracks currently-held symbols
- Pre-trade check: if adding a NEW symbol would exceed `max_open_positions`, deny
- Adding to an existing position is allowed (doesn't increase symbol count)

### Notional value cap

**Implementation:**
- Per-order: `qty * price <= max_position_value`
- Note: this only fires when `price` is provided (LIMIT orders). MARKET orders bypass — for those, the position-count and daily-loss caps are the actual ceilings.

## What this codebase does NOT enforce automatically

Some SEBI rules need human discipline or external tooling:

- **Algo registration with the exchange** — for personal-scale automation under a personal API account, exchange registration is generally not required, but verify your broker's interpretation. If you're considered "execution algo" (auto-executing orders without human approval per order), some brokers require additional disclosures.
- **Strategy backtesting evidence** — keep your backtest reports outside the codebase. Required if a regulator ever audits.
- **Algo ID tagging** — Kite's `tag` field (max 20 chars) is used in `broker_adapter.py` (`tag = f"strat:{self.id}"`). Verify this is sufficient for your broker; some require a registered algo ID.
- **Co-location disclosure** — running on AWS ap-south-1 is not co-location. If you ever move to NSE colo, separate filings apply.

## Verification checklist before going live

- [ ] NAT GW EIP registered with broker (`terraform output whitelist_ip`)
- [ ] Test order from execution node — confirm it lands at the broker without IP rejection
- [ ] TOTP seed in Secrets Manager, manually verified by running the auth refresh once
- [ ] Rate limiter tested: send 15 orders in 1 second to a non-prod broker account, verify 5 are 429'd
- [ ] Kill switch tested: engage → place test order → confirm rejection → release
- [ ] Audit log shows entries for both successful and rejected orders
- [ ] Daily P&L cap tested with a small `max_daily_loss` value
- [ ] CloudWatch alarm on `auth_refresh` Lambda errors → SNS → email
- [ ] CloudWatch alarm on ingestor `kite.disconnected` log pattern → SNS → email
